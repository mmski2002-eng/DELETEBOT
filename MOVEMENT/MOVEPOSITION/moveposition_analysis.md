## **ШАГ 1: ИДЕНТИФИКАЦИЯ ЯДРА АРХИТЕКТУРЫ**

### 1.1 Высокоуровневая модель

Протокол реализует двухуровневую архитектуру:

| Уровень | Компонент | Назначение |
|---------|-----------|------------|
| **Смарт-контракты (on-chain)** | **Broker** (Move-модуль) | Изолированный пул ликвидности для одного или группы активов. Содержит логику лэндинга, заимствования, расчёта процентов и ликвидаций |
| **Off-chain сервис (Super-API)** | `superposition-stable-porto` | Google Cloud Run сервис, предоставляющий REST API: агрегация цен, расчёт риска, история, симуляции, рефералы |

### 1.2 Модель «Broker»

Каждый **Broker** — это независимый смарт-контракт (аккаунт в Movement), реализующий изолированный денежный рынок. Ключевые свойства:

- **Изоляция риска**: каждый Broker управляет собственным набором залоговых активов и параметров риска. Банкротство одного Broker'а не затрагивает другие
- **Адресация**: Broker идентифицируется по `networkAddress` (адрес в Movement) и внутреннему `id`
- **Guardrails**: каждый Broker имеет встроенные ограничения (`CoreTypes_BrokerGuardrail`), задающие лимиты LTV, liquidation threshold, максимальный объём заимствований и т.д.

### 1.3 Ключевые сущности SDK (TypeScript-модели)

Из OpenAPI-клиента извлечены следующие структуры данных:

**Рыночные данные**:
- `FullInstrument` / `RawInstrument` — описание актива (название, адрес, decimals, логотип)
- `PricesResponse` — текущие цены
- `CoreTypes_HistoricalPrices` — исторические цены (годовая глубина)
- `InterestRateCurve` — кривая процентных ставок
- `CoreTypes_RiskParamsHistory` — исторические параметры риска

**Портфель и позиции**:
- `CoreTypes_PortfolioState` — состояние портфеля пользователя (активы, долги)
- `CoreTypes_PortfolioWithFullRisk` — портфель с полным расчётом риска
- `CoreTypes_RiskEval` — оценка риска (health factor и т.д.)
- `PositionSuperCore` — позиция пользователя в конкретном Broker'е

**Транзакции**:
- `TransactionRequest` — запрос на операцию (lend, borrow, redeem, repay)
- `PacketResponse` — «хокет-пакет» (подписанная транзакция для отправки в сеть)
- `CoreTypes_BrokerHistory` — история операций внутри Broker'а
- `CoreTypes_EventTransaction` / `CoreTypes_FullEventTransactions` — полная история событий

**Прочее**:
- `CoreTypes_Referal` — реферальная программа
- `CoreTypes_UserStrategy` / `CoreTypes_FlashLoanStrategy` — стратегии (включая flash loans)
- `WalletBalances` — баланс кошелька

## **ШАГ 2: АНАЛИЗ ФИНАНСОВЫХ ПРИМИТИВОВ**

### 2.1 Базовые операции

Протокол поддерживает полный цикл кредитования:

| Операция | API Endpoint | Описание |
|----------|-------------|-----------|
| **Lend** | `POST /brokers/lend/v2` | Предоставление ликвидности в Broker, получение процентного дохода |
| **Redeem** | `POST /brokers/redeem/v2` | Вывод ликвидности из Broker'а (обратная операция к lend) |
| **Borrow** | `POST /brokers/borrow/v2` | Заимствование активов под залог ранее внесённой ликвидности |
| **Repay** | `POST /brokers/repay/v2` | Погашение займа (обратная операция к borrow) |

### 2.2 Процентная модель

Сущность `InterestRateCurve` указывает на **динамическую модель процентных ставок** — ставки по займам и депозитам зависят от коэффициента использования пула (utilization rate). Типичная реализация в Move: кусочно-линейная кривая с точками перегиба.

### 2.3 Flash Loans

Наличие типа `CoreTypes_FlashLoanStrategy` подтверждает поддержку **мгновенных займов без залога** (flash loans). Такие займы должны быть возвращены в пределах одной транзакции.

### 2.4 Ликвидации

Два эндпоинта для ликвидаций:
- `POST /brokers/liquidate/v1` — ликвидация по версии 1
- `POST /brokers/liquidate/v2` — ликвидация по версии 2

Наличие версионирования указывает на эволюцию логики ликвидаций (возможно, частичная ликвидация vs полная, или разные механизмы ценообразования бонуса ликвидатора).

## **ШАГ 3: УПРАВЛЕНИЕ РИСКАМИ**

### 3.1 Модель расчёта риска

Off-chain сервис предоставляет выделенный эндпоинт:

- `GET /risk/full/{address}` — возвращает `CoreTypes_PortfolioWithFullRisk` (полный портфель с агрегированной оценкой риска)
- `GET /risk/inputs/{address}` — возвращает `CoreTypes_RiskInput` (входные параметры для расчёта риска)
- `POST /risk/simulated` — симуляция риска на произвольном состоянии портфеля
- `PUT /risk/custom` — пользователь может задать собственные параметры риска и получить оценку

### 3.2 Ключевые риск-параметры

Из моделей:
- **LTV (Loan-to-Value)**: максимальная доля залога, доступная для заимствования
- **Liquidation Threshold**: порог, при котором позиция может быть ликвидирована
- **Health Factor**: агрегированный показатель «здоровья» портфеля (вероятно, рассчитывается как взвешенное отношение залога к долгу)

### 3.3 Guardrails

Каждый Broker имеет набор встроенных ограничений (`CoreTypes_BrokerGuardrail`):
- Максимальный объём депозитов
- Максимальный объём заимствований
- Минимальный/максимальный LTV
- Ограничения по типам активов

### 3.4 Мониторинг риска

- `GET /risk/params/history` — исторические параметры риска (позволяет отслеживать изменения риск-политики протокола)
- `GET /prices/full` — полная история цен по всем инструментам
- `GET /toggles` / `GET /toggles/{toggleName}` — feature-флаги (позволяют администраторам включать/отключать функции протокола)

## **ШАГ 4: МОДЕЛЬ ЦЕНООБРАЗОВАНИЯ И ОРАКУЛЫ**

### 4.1 Источники цен

Цены предоставляются off-chain сервисом через эндпоинты:
- `GET /prices?names[]=...` — текущие цены
- `GET /prices/historical/year` — исторические цены за год
- `GET /prices/full` — детальная информация по всем инструментам

**Вывод**: используется **гибридная оракульная модель**, где off-chain сервис агрегирует цены с различных источников (вероятно, Pyth, Switchboard или собственные ноды), а on-chain контракты получают верифицированные цены через механизм подписи.

### 4.2 Типы ценовых данных

- `PricesResponse` — текущие цены (Spot)
- `HistoricalPrice` — историческая цена с временной меткой
- `CoreTypes_PricedInstrument` — актив с привязанной ценой
- `CoreTypes_FullInstrumentsResponse` — полный каталог инструментов с ценами, логотипами, метаданными

## **ШАГ 5: ОЦЕНКА ЛИКВИДНОСТИ И TVL**

**Ограничение**: прямые данные о TVL (Total Value Locked) не доступны через исследованные SDK и API без авторизации/доступа к бэкенду.

**Косвенные индикаторы**:

1. **Наличие отдельного Explorer** (explorer.movementnetwork.xyz) и индексера (GraphQL endpoint) указывает на зрелый уровень инфраструктуры, необходимый для работы с реальной ликвидностью
2. **Google Cloud Run деплой** API-сервиса указывает на production-окружение с автоматическим масштабированием
3. **Версионирование эндпоинтов** (v1, v2) свидетельствует о работающем продукте с историей обновлений
4. **Реферальная программа** — маркер активного привлечения пользователей и роста TVL

**Рекомендация**: для получения точных данных о TVL следует:
- Запросить все Broker'ы через `GET /brokers` и агрегировать балансы
- Использовать Movement Indexer GraphQL API для прямого запроса балансов счетов
- Обратиться к Movement Explorer для визуального анализа

## **ШАГ 6: БЕЗОПАСНОСТЬ И ДЕЦЕНТРАЛИЗАЦИЯ**

### 6.1 Модель безопасности

| Аспект | Оценка | Обоснование |
|--------|--------|------------|
| **Изоляция пулов** | ✅ Позитивно | Изолированные Broker'ы предотвращают каскадные банкротства |
| **Off-chain зависимость** | ⚠️ Внимание | Off-chain сервис для расчёта риска и цен — централизованная точка отказа |
| **Feature-флаги** | ⚠️ Внимание | Централизованный механизм включения/отключения функций |
| **Аудит** | ❓ Неизвестно | Публичные отчёты об аудите не обнаружены |

### 6.2 Ключевой вывод

Протокол MovePosition использует **гибридную архитектуру**:
- On-chain: изолированные пулы, логика займов, процентов и ликвидаций — децентрализовано
- Off-chain: агрегация цен, расчёт риска, симуляции — централизованный сервис

Такой подход обеспечивает **гибкость и скорость разработки**, но создаёт зависимость от uptime и корректности работы off-chain сервиса.

## **ШАГ 7: ПОЛЬЗОВАТЕЛЬСКИЙ ОПЫТ И ИНТЕГРАЦИЯ**

### 7.1 Модель взаимодействия

Пользователи взаимодействуют с протоколом **опосредованно** через SDK:

1. Пользователь формирует запрос через TypeScript SDK
2. SDK отправляет запрос в Super-API
3. API возвращает подписанный «пакет» (PacketResponse)
4. Пользователь отправляет пакет в сеть Movement через wallet

### 7.2 API-эндпоинты для пользователей

| Категория | Эндпоинты | Описание |
|-----------|-----------|----------|
| Брокеры | `/brokers`, `/brokers/address/*`, `/brokers/guardrails` | Информация о доступных рынках |
| Портфель | `/portfolio/{address}`, `/portfolio/{address}/history`, `/portfolio/{address}/cumulativeHistory` | Состояние и история портфеля |
| Операции | `/brokers/lend/v2`, `/brokers/borrow/v2`, `/brokers/redeem/v2`, `/brokers/repay/v2`, `/brokers/liquidate/v2` | Все финансовые операции |
| Риск | `/risk/full/{address}`, `/risk/simulated`, `/risk/custom`, `/risk/inputs/{address}` | Оценка и симуляция риска |
| Кошельки | `/wallets/{address}` | Балансы кошелька (более не используется — deprecated?) |
| Рефералы | `/referals/*` | Управление реферальными кодами |

### 8.2 Сильные стороны
- Изолированная модель Broker'ов (security by isolation)
- Полный набор DeFi-примитивов
- Хороший TypeScript SDK для разработчиков
- Реферальная программа для роста
- Feature-флаги для контролируемых обновлений
- Production-деплой на Google Cloud Run

### 8.3 Зоны риска
- Зависимость от off-chain сервиса для критических функций
- Отсутствие публичного аудита смарт-контрактов
- Непрозрачность источников цен оракулов
- Централизованное управление feature-флагами

### 8.4 Рекомендации
1. **Провести независимый аудит** смарт-контрактов с публикацией результатов
2. **Децентрализовать оракулы** — осуществить миграцию на полностью on-chain решения (Pyth, Switchboard)
3. **Получить данные по TVL** через прямые запросы к Movement Indexer для количественной оценки ликвидности
4. **Внедрить мультиподпись** (multisig) для управления feature-флагами
5. **Мониторить параметр Health Factor** всех активных позиций как опережающий индикатор каскадных ликвидаций
